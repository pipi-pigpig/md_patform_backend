# 1 项目概述

## 1.1 项目背景

在新能源锂电池研发领域，电解液作为电池核心组分，其理化性质（密度、电导率、粘度、介电常数等）与微观溶剂化结构直接决定了电池的循环寿命、倍率性能与安全特性。传统电解液研发依赖大量试错实验，研发周期长、成本高；而分子动力学（MD）模拟可从原子层面解析电解液微观行为，预测宏观理化性质，已成为电解液配方优化的核心技术手段。

主流MD模拟工具（如LAMMPS）存在使用门槛高、操作流程复杂、环境部署困难、结果后处理繁琐等问题，要求使用者具备深厚的计算化学、分子动力学理论基础与编程能力，难以被电池研发工程师直接使用。为此，本项目旨在打造一款面向电解液研发场景、全流程自动化、低使用门槛的在线分子动力学计算平台，实现电解液配方从建模、模拟计算、后处理到结果可视化的全链路闭环，降低MD模拟技术的应用门槛，加速电解液材料的研发进程。

## 1.2 项目目标

### 1.2.1 核心目标

1. 构建可视化Web操作界面，实现电解液配方参数化输入，替代传统手动编写LAMMPS脚本的操作模式；
2. 实现电解液体系自动化建模，集成Moltemplate、Packmol工具，完成分子拓扑构建、初始构型随机堆积，一键生成LAMMPS可识别的输入文件；
3. 基于Docker容器化技术封装LAMMPS计算引擎，实现模拟任务的自动化调度、并行执行与状态实时监控；
4. 集成MDSuite专业后处理工具，自动完成轨迹文件解析与目标性质计算，输出标准化的理化性质结果；
5. 实现计算结果的二维图表可视化与三维分子结构动态渲染，直观展示电解液微观结构与宏观性质变化。

## 1.3 项目范围

本平台聚焦于锂电池电解液体系的经典分子动力学模拟，核心覆盖电解液体系构建、能量最小化、系综平衡、生产模拟、性质计算与结果可视化全流程，支持密度、电导率、粘度、介电常数、溶剂化结构等核心理化性质的自动化计算。平台采用B/S架构，用户通过浏览器即可完成全流程操作，无需在本地部署任何MD模拟环境。

# 2 系统总体架构设计

本平台采用分层架构与容器化部署模式，整体架构从上至下分为6层，分别为：前端展示层、网关接入层、应用服务层、计算引擎层、数据存储层、基础设施层，同时配套跨层的安全体系与监控运维体系。

## 2.1 分层架构设计

### 2.1.1 前端展示层

前端展示层是用户与平台交互的唯一入口，采用Vue3+Element Plus构建单页应用（SPA），为用户提供可视化操作界面，核心包含三大功能视图：

1. **工作台视图**：展示用户任务概览、系统状态、快捷操作入口；
2. **电解液配方管理视图**：提供配方创建、编辑、模板管理、分子库查询功能；
3. **模拟任务管理视图**：提供任务创建、状态监控、日志查看、结果可视化功能。

前端集成Plotly.js与3Dmol.js可视化引擎，完成计算结果的二维图表与三维分子结构渲染，通过Axios封装的API客户端与后端服务进行数据交互。

### 2.1.2 网关接入层

网关接入层基于Nginx实现，核心承担两大能力：

1. **静态资源服务**：托管前端打包后的静态文件，提供静态资源访问加速；
2. **反向代理**：将前端/api路径的请求统一转发至后端Spring Boot服务，解决跨域问题。

### 2.1.3 应用服务层

应用服务层是平台的业务核心，基于Spring Boot 3框架开发，采用分层设计模式，拆解为Controller控制层、Service业务服务层、Mapper数据访问层，封装平台所有业务逻辑，核心服务模块包括：

- **用户认证与权限服务**：处理用户登录、身份校验、权限管理；
- **模拟任务管理服务**：处理任务创建、状态更新、队列管理、生命周期控制；
- **模板引擎服务**：基于Jinja2实现LAMMPS输入脚本的动态生成；
- **Docker调度服务**：基于Docker Java API实现计算容器的管理与命令执行；
- **MD执行服务**：封装LAMMPS模拟执行全流程，处理模拟输入/输出；
- **后处理分析服务**：集成MDSuite，实现轨迹文件解析与理化性质自动化计算；
- **文件管理服务**：处理用户文件上传、下载、目录管理、结果文件归档。

### 2.1.4 计算引擎层

计算引擎层是平台的算力核心，基于Docker容器化技术实现，所有计算组件均封装在独立容器中，与宿主机环境解耦，核心包含核心引擎封装、容器化调度、执行容错、电解液配方管理、自动化建模、后处理分析等能力，具体组件如下：

1. **LAMMPS计算容器**：预编译LAMMPS软件、MPI并行环境、CUDA GPU加速支持，是分子动力学模拟的核心执行单元；
2. **建模工具集**：容器内预置Moltemplate、Packmol工具，负责电解液体系的自动化建模与初始构型生成；
3. **后处理工具集**：容器内预置MDSuite、MDAnalysis、NumPy、SciPy等Python分析库，负责轨迹文件的后处理与性质计算；
4. **可视化工具集**：预置Plotly、3Dmol.js的后端渲染依赖，负责可视化数据的预处理。

所有容器通过Docker网桥与后端服务通信，通过宿主机目录卷挂载实现输入/输出文件的共享。

### 2.1.5 数据存储层

数据存储层采用多存储组件融合的架构，针对不同类型的数据选择适配的存储介质，核心包括：

1. **MySQL 8.0主数据库**：存储结构化业务数据，包括用户信息、电解液配方、模拟任务元数据、计算结果数值、系统配置等；
2. **文件系统**：存储非结构化文件数据，包括LAMMPS输入/输出文件、MD轨迹文件、可视化渲染结果文件；
3. **Moltemplate分子库**：存储预置的电解液分子模板文件（.lt格式）、力场参数文件，为自动化建模提供基础数据支撑。

### 2.1.6 基础设施层

基础设施层是平台运行的底层支撑，包括服务器硬件资源、操作系统、Docker容器运行时、网络环境、GPU驱动/CUDA运行库等，为上层服务提供算力、存储、网络基础能力。

## 2.2 物理部署架构

平台采用Docker Compose实现一站式容器化部署，所有服务组件均以容器方式运行，通过自定义网桥实现网络互通，通过数据卷实现数据持久化。核心部署组件与网络架构如下表所示：

### 容器信息表

| 容器名称 | 核心功能 | 端口映射 | 依赖服务 | 数据卷挂载 |
|----------|----------|----------|----------|------------|
| electrolyte_frontend | Nginx+Vue3前端 | 80:80 | backend | ./frontend/dist:/usr/share/nginx/html |
| electrolyte_backend | Spring Boot后端服务 | 8080:8080 | mysql, md-engine | ./data/uploads:/app/uploads；./data/results:/app/results；/var/run/docker.sock:/var/run/docker.sock |
| electrolyte_mysql | MySQL 8.0数据库 | 3307:3306 | 无 | ./data/mysql:/var/lib/mysql |

# 5 数据库设计概要

## 5.1 整体存储架构

平台采用"结构化数据+非结构化数据"分离的存储架构，整体设计如下：

1. **结构化数据**：全部存入MySQL 8.0数据库，采用InnoDB存储引擎，支持事务与行级锁，保证数据一致性；
2. **非结构化数据**：包括分子结构文件、轨迹文件、模拟输入/输出文件、图片/视频等，存入宿主机文件系统，MySQL中仅存储文件访问路径与元数据；
3. **静态模板数据**：包括分子.lt模板、力场参数文件，存入专用的分子库目录，通过文件路径索引访问。

## 7.2 核心接口类定义

### 7.2.1 计算引擎服务接口（ComputeEngineService）

**接口定义**：常规功能模块通过此接口调用计算引擎功能。

```java
package com.mdplatform.service.compute;

/**
 * 计算引擎服务接口
 * 定义常规功能模块与配方及计算引擎功能模块之间的交互契约
 */
public interface ComputeEngineService {
```

## 7.3 接口数据结构定义

### 7.3.0 类包分类说明

根据项目包结构规范，类按照用途分为以下两类：

| 包路径 | 用途 | 特征 | 示例类 |
|--------|------|------|--------|
| `com.mdplatform.model` | 实体类（Entity） | 使用`@Entity`注解，对应数据库表，持久化存储 | `SimulationJob`, `ElectrolyteSystem`, `SysUser`, `MoleculeTemplate`, `CalculationResult`, `JobExecutionLog` |
| `com.mdplatform.dto` | 数据传输对象（DTO） | 无JPA注解，用于API请求/响应、前后端数据交换 | `SystemDto`, `SimulationDto`, `JobSubmitRequest`, `JobStatusResponse`, `ComputeResultResponse`, `FormulaCreateRequest`, `FormulaCreateResponse`, `FormulaUpdateRequest`, `FormulaQueryRequest`, `FormulaListResponse`, `FormulaDetailResponse`, `MoleculeTemplateListResponse`, `MoleculeTemplateDetailResponse` |

**分类原则**：
- **model包**：对应数据库表的实体类，包含`@Entity`、`@Table`、`@Id`等JPA注解
- **dto包**：用于数据传输的类，不包含JPA注解，通常有`fromEntity()`方法将实体转换为DTO

### 7.4.1 任务管理服务（常规功能模块）

常规功能模块中的任务服务通过依赖注入使用计算引擎接口：

```java
package com.mdplatform.service.task;

import com.mdplatform.service.compute.ComputeEngineService;
import com.mdplatform.dto.compute.*;
import org.springframework.stereotype.Service;
import org.springframework.beans.factory.annotation.Autowired;

/**
 * 模拟任务管理服务（常规功能模块）
 * 通过ComputeEngineService接口调用计算引擎功能
 */
@Service
public class SimulationTaskService {
    
    @Autowired
    private ComputeEngineService computeEngineService;
    
    @Autowired
    private TaskRepository taskRepository;
    
    /**
     * 提交新的模拟任务
     */
    public JobSubmitResponse submitTask(TaskSubmitDto submitDTO, String userId) {
        // 1. 构建任务提交请求
        JobSubmitRequest request = buildJobRequest(submitDTO);
        
        // 2. 调用计算引擎接口提交任务
        JobSubmitResponse response = computeEngineService.submitJob(request);
        
        // 3. 保存任务记录到数据库（使用实体类）
        SimulationJob job = new SimulationJob();
        job.setJobId(Long.parseLong(response.getJobId()));
        job.setJobName(submitDTO.getJobName());
        job.setUserId(Long.parseLong(userId));
        job.setStatus("PENDING");
        taskRepository.save(job);
        
        // 4. 注册状态监听器
        computeEngineService.registerStatusListener(
            response.getJobId(), 
            new TaskStatusUpdateListener(this)
        );
        
        return response;
    }
    
    /**
     * 查询任务状态
     */
    public JobStatusResponse queryTaskStatus(String jobId) {
        return computeEngineService.getJobStatus(jobId);
    }
    
    /**
     * 取消任务
     */
    public OperationResult cancelTask(String jobId) {
        return computeEngineService.cancelJob(jobId);
    }
    
    /**
     * 获取任务结果
     */
    public ComputeResultResponse getTaskResults(String jobId, List<String> properties) {
        return computeEngineService.getComputeResults(jobId, properties);
    }
    
    /**
     * 任务状态更新回调处理
     */
    public void onStatusUpdate(String jobId, JobStatus newStatus, double progress) {
        SimulationJob job = taskRepository.findByJobId(Long.parseLong(jobId));
        job.setStatus(newStatus.name());
        taskRepository.save(job);
        
        // 通过WebSocket推送给前端
        webSocketHandler.pushStatusUpdate(jobId, newStatus, progress);
    }
    
    // 私有方法：构建任务请求
    private JobSubmitRequest buildJobRequest(TaskSubmitDto dto) {
        JobSubmitRequest request = new JobSubmitRequest();
        request.setJobName(dto.getJobName());
        request.setSystemId(dto.getSystemId());
        request.setTargetProperties(dto.getTargetProperties());
        request.setSimulationParams(dto.getSimulationParams());
        request.setHardwareConfig(dto.getHardwareConfig());
        return request;
    }
    
    /**
     * 创建电解液配方
     */
    public FormulaCreateResponse createFormula(FormulaCreateRequest request) {
        return computeEngineService.createFormula(request);
    }
    
    /**
     * 编辑电解液配方
     */
    public OperationResult updateFormula(FormulaUpdateRequest request) {
        return computeEngineService.updateFormula(request);
    }
    
    /**
     * 删除电解液配方

    }
```

## 7.6 接口安全机制

### 7.6.1 身份认证

1. 基于JWT实现无状态身份认证，Token有效期24小时；
2. 所有接口请求需携带有效Token，否则返回401未授权；

### 7.6.2 权限控制

1. 基于RBAC模型实现接口级权限控制；
2. 不同角色用户可访问的接口集合不同；
3. 禁止越权访问其他用户的数据。

### 7.6.3 参数校验

1. 所有接口输入参数进行严格校验；
2. 防止SQL注入、XSS跨站脚本、命令注入等安全漏洞；
3. 文件上传接口校验文件类型与大小。

### 7.6.4 通信安全

1. 平台所有接口通信采用HTTPS协议；
2. 配置SSL证书，防止数据传输过程中被窃听、篡改；
3. 严格配置CORS跨域规则，仅允许可信域名访问。

# 8 核心业务流程概要

本平台的核心业务流程围绕"用户配方输入→自动化建模→模拟计算→后处理分析→结果可视化"全链路展开，主要包含以下子流程：

## 8.1 主业务全流程

1. 用户完成平台登录，创建/选择电解液配方，提交模拟任务；
2. 后端服务校验参数，生成任务ID，存入数据库，初始状态为PENDING；
3. 自动化体系构建：调用Moltemplate与Packmol完成分子拓扑组装与初始构型生成；
4. 输入脚本生成：基于Jinja2模板引擎动态生成LAMMPS输入脚本；
5. 计算资源准备：检查md-engine容器状态，传入输入文件；
6. LAMMPS模拟执行：在容器内执行能量最小化→NVT平衡→NPT平衡→生产模拟；
7. 任务状态实时更新：实时解析模拟日志，更新任务状态为RUNNING；
8. 模拟结果回收：将输出文件复制到宿主机，更新任务状态为COMPLETED；
9. 自动化后处理：调用MDSuite完成目标性质的自动化计算；
10. 结果数据存储：将计算结果存入数据库与文件系统；
11. 结果可视化渲染：前端调用Plotly.js与3Dmol.js完成可视化；
12. 结果查看与导出：用户查看计算结果，支持下载导出。

## 8.2 Moltemplate自动化建模子流程

1. 解析配方参数，计算模拟盒子尺寸与各分子数量；
2. 从分子库调取单分子结构文件与Moltemplate模板文件；
3. 调用Packmol完成分子随机堆积，生成初始构型文件；
4. 生成Moltemplate系统描述文件system.lt；
5. 执行Moltemplate命令，生成LAMMPS输入文件包。

## 8.3 LAMMPS模拟计算子流程

1. 模拟初始化：加载结构文件，设置力场参数；
2. 初始速度分配：根据温度分配Maxwell-Boltzmann分布；
3. 能量最小化：消除不合理原子重叠；
4. NVT平衡模拟：恒温平衡200ps；
5. NPT平衡模拟：恒温恒压平衡500ps；
6. 生产模拟：执行用户指定时长的模拟，输出目标性质数据。

## 8.4 MDSuite后处理分析子流程

1. 读取任务配置，获取目标计算性质；
2. 加载LAMMPS输出文件，完成格式转换；
3. 执行轨迹预处理：坐标解包裹；
4. 分性质执行自动化计算：密度、电导率、粘度、介电常数、溶剂化结构；
5. 生成标准化JSON格式结果数据。

# 10 项目总结

本方案设计的基于分子动力学的电解液计算平台，深度贴合锂电池电解液研发的业务场景，解决了传统MD模拟工具使用门槛高、操作复杂的行业痛点。平台通过可视化Web界面+自动化建模+容器化计算引擎+智能化后处理的技术方案，实现了电解液分子动力学模拟全流程的自动化与标准化，让电池研发人员无需掌握深厚的计算化学知识，即可完成电解液配方的MD模拟与性质预测。

平台采用前后端分离的分层架构与Docker容器化部署模式，具备良好的可扩展性、可维护性与安全性，核心功能模块低耦合高内聚，支持后续计算引擎、分析算法、分子模板库的快速扩展。同时，平台基于行业标准的模拟流程与分析算法，保证计算结果的准确性与可靠性，能够有效辅助电解液配方的研发与优化，缩短研发周期，降低实验成本。
